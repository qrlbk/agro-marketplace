# Модель угроз (Threat Model)

Краткий документ для аудита безопасности agro-marketplace: роли, активы и основные классы угроз.

---

## 1. Архитектура системы (high-level)

- **Backend:** FastAPI API (PostgreSQL, Redis), JWT-аутентификация, OTP по SMS, staff-логин по паролю, вебхуки 1C, интеграция с OpenAI, загрузки файлов поставщиков (ClamAV).
- **Frontend:** React 18 + TypeScript SPA (клиентское приложение + Staff portal).
- **Инфраструктура:** Docker (Postgres, Redis), CI в GitHub Actions.

---

## 2. Роли и акторы

| Роль | Описание | Канал доступа |
|------|----------|----------------|
| **Пользователь (покупатель)** | Регистрация по OTP, каталог, корзина, заказы, гараж, чат, отзывы, профиль | Web (JWT access/refresh) |
| **Поставщик (vendor)** | Пользователь, привязанный к компании со статусом vendor; загрузка файлов, заказы по своей компании | Web (тот же JWT, проверка по company/vendor_id) |
| **Staff** | Сотрудник поддержки/операций; отдельный логин (логин/пароль), Staff portal, ограниченные права по permissions | Web (Staff JWT) |
| **Admin (marketplace)** | Пользователь с ролью `admin` или Staff с админскими правами; полный доступ к данным, категории, аудит | Web (JWT / Staff JWT) |
| **Интеграция 1C** | Внешняя система; отправляет вебхуки (заказы, синхронизация) | API (X-API-Key: WEBHOOK_1C_API_KEY) |
| **SMS-шлюз** | Исходящие SMS (OTP); вызывается backend’ом | Исходящие запросы (Bearer token) |
| **OpenAI** | LLM для чата, нормализация цен; вызывается backend’ом | Исходящие запросы (API key) |

**Внешний атакующий** — неаутентифицированный или подставной пользователь (в т.ч. скомпрометированный аккаунт).

## 3. Активы (что защищаем)

| Актив | Описание | Критичность |
|-------|----------|-------------|
| Учётные записи пользователей | user_id, phone, профиль, onboarding, consent (chat_storage_opt_in) | Высокая |
| Учётные записи Staff | staff_id, login, password_hash, permissions, привязка к компаниям | Критическая |
| Заказы и финданные | Заказы, разбивка по поставщикам, адреса, комментарии | Высокая |
| Файлы поставщиков | Загруженные документы/изображения в хранилище vendor | Средняя (репутация, malware) |
| Токены и ключи | JWT secrets, refresh secret, staff JWT secret, OTP secret, WEBHOOK_1C_API_KEY, OPENAI_API_KEY, HEALTH_API_KEY, SMS API key | Критическая |
| База данных | PostgreSQL: пользователи, заказы, компании, чаты, уведомления, аудит | Критическая |
| Логи и аудит | Audit log, логи приложения, логи LLM (при включённом хранении) | Средняя (конфиденциальность, PII) |
| Конфигурация | .env, переменные окружения, настройки CORS, CSP, rate limits | Высокая |

## 4. Классы угроз и сценарии

### 4.1 Компрометация учётной записи

- **Брутфорс OTP / демо-логина:** перебор кодов или пароля при слабых rate limit или их обходе.
- **Брутфорс Staff-логина:** перебор паролей; слабый или утёкший дефолтный пароль из миграции 008.
- **Кража/подмена JWT:** утечка access/refresh токена (localStorage, перехват, XSS); использование украденного refresh без ротации или без blacklist.
- **Слабые секреты:** короткие или предсказуемые JWT_SECRET / STAFF_JWT_SECRET позволяют подделывать токены.

### 4.2 IDOR (Insecure Direct Object Reference)

- Подстановка чужого `order_id`, `user_id`, `garage_id`, `session_id` (чат), `notification_id` и т.д. для доступа к чужим данным при отсутствии проверки владельца или прав.

### 4.3 Эскалация привилегий

- Пользователь получает доступ к маршрутам Staff/Admin при пропущенной проверке `get_current_staff` / `require_admin_or_staff`.
- Подмена `role` или `permissions` в JWT при некорректной верификации или при использовании «none» алгоритма.
- Обход проверок через альтернативные эндпоинты или неконсистентную проверку прав между роутерами.

### 4.4 Инъекции

- **SQLi:** конкатенация пользовательского ввода в SQL (в т.ч. в raw SQL в миграциях или в admin/audit).
- **NoSQLi:** не применимо (БД — PostgreSQL).
- **XSS:** отображение непросанитизированного пользовательского ввода в HTML (dangerouslySetInnerHTML, innerHTML).
- **Инъекция в LLM:** промпт-инъекция для утечки системного промпта или выполнения несанкционированных действий.

### 4.5 CSRF

- Выполнение критичных действий (смена профиля, создание заказа, отправка обращений) по запросу с другого сайта при отсутствии CSRF-токенов или проверки Origin/Referer для state-changing операций.

### 4.6 SSRF

- Вызов внешних URL из backend (например, по данным из вебхука или из профиля) без валидации/белого списка хостов.

### 4.7 Загрузка файлов и RCE

- Загрузка исполняемых файлов, скриптов или файлов с опасными именами (path traversal) при слабой валидации расширений/MIME или при отключённом/обходе ClamAV.
- DoS через исчерпание квоты или размера запроса.

### 4.8 Утечка секретов и информации

- Секреты в коде, в логах, в ответах API или в фронтенд-сборке.
- Раскрытие внутренних ошибок или стека в ответах.
- Health-эндпоинты без HEALTH_API_KEY раскрывают состояние БД/Redis/OpenAI.

### 4.9 Атаки на интеграции

- **Вебхук 1C:** поддельные запросы при утёкшем или слабом WEBHOOK_1C_API_KEY; повторное использование запросов (replay) при отсутствии защиты.
- **SMS:** перерасход при отсутствии rate limit на запрос OTP; утечка токена SMS в логах.
- **OpenAI:** утечка ключа; логирование PII в промптах/ответах при неверной настройке chat_store_enabled / chat_prompt_log_level.

### 4.10 DoS и злоупотребление ресурсами

- Исчерпание rate limit или обход лимитов (распределённые запросы, разные IP).
- Тяжёлые запросы (большие тела, сложные запросы к БД, длинные чат-сессии) без лимитов.
- Исчерпание хранилища поставщика или диска через массовые загрузки.

---

## 5. Доверительные границы

- **Периметр API:** все входящие запросы к FastAPI считаются недоверенными до успешной аутентификации и проверки авторизации.
- **Пользовательские данные:** любой ввод (тело запроса, query, path, заголовки от клиента) не доверяется без валидации и санитизации.
- **Вебхуки и интеграции:** источник считается легитимным только после проверки ключа/подписи; тело вебхука — недоверенное.
- **Frontend:** не является доверительной границей для авторизации; все проверки доступа выполняются на backend.

## 6. Связь с этапами аудита

- **Этап 2 (конфиг и секреты):** активы «Токены и ключи», «Конфигурация».
- **Этапы 3–4 (auth, ввод, загрузки):** угрозы 4.1–4.3, 4.7, 4.8.
- **Этап 5 (БД, RLS):** актив «База данных», угрозы IDOR, SQLi, эскалация.
- **Этапы 7–8 (frontend, infra):** угрозы XSS, утечка секретов, DoS.
- **Этапы 9–10 (зависимости, динамические тесты):** все классы угроз в рамках автоматизированной и ручной проверки.

Этот документ можно расширять по мере появления новых ролей, интеграций и активов.
